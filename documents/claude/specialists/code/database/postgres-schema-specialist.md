**Pattern** | パターン | Mẫu:
```sql
-- StarX4CRM Standard Table Structure
-- (matches AbstractAuditingEntity<Long>)
CREATE TABLE cmn_m_customer (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,

    -- business columns --
    customer_code VARCHAR(50) NOT NULL UNIQUE,
    customer_name VARCHAR(255) NOT NULL,
    email VARCHAR(255),
    status VARCHAR(50) NOT NULL DEFAULT 'ACTIVE',

    -- standard audit columns (MANDATORY for all tables) --
    ins_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    ins_user_id BIGINT NOT NULL,
    upd_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    upd_user_id BIGINT,
    upd_cnt INTEGER NOT NULL DEFAULT 1,       -- optimistic locking version
    del_flg BOOLEAN NOT NULL DEFAULT false,    -- soft delete flag
    del_date TIMESTAMP,
    del_user_id BIGINT,

    -- check constraints --
    CONSTRAINT chk_cmn_m_customer_status CHECK (status IN ('ACTIVE', 'INACTIVE', 'SUSPENDED'))
);

-- indexes on frequently queried columns
CREATE INDEX idx_cmn_m_customer_code ON cmn_m_customer(customer_code);
CREATE INDEX idx_cmn_m_customer_status ON cmn_m_customer(status);
CREATE INDEX idx_cmn_m_customer_del_flg ON cmn_m_customer(del_flg) WHERE del_flg = false;

COMMENT ON TABLE cmn_m_customer IS 'Customer master table for StarX4CRM';
```

|--------|-----------|---------|-------------|

| `ins_user_id` | Long | BIGINT | Creator user ID |

| `upd_user_id` | Long | BIGINT | Last modifier user ID |

| `del_flg` | Boolean | BOOLEAN | Soft delete flag |

| `del_user_id` | Long | BIGINT | Deleter user ID |

**Why Approved** | 承認理由 | Lý Do Chấp Nhận:

#### 2. Foreign Key with Cascading Rules | カスケードルール付き外部キー | Khóa Ngoại với Quy Tắc Cascade

**Evidence**: PostgreSQL Foreign Key Documentation

**Pattern** | パターン | Mẫu:
```sql
CREATE TABLE sfa_t_opportunity (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer_id BIGINT NOT NULL,
    sales_user_id BIGINT NOT NULL,
    amount DECIMAL(15, 2) NOT NULL,
    status VARCHAR(50) NOT NULL DEFAULT 'OPEN',
    probability INTEGER DEFAULT 0,

    -- audit columns --
    ins_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    ins_user_id BIGINT NOT NULL,
    upd_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    upd_user_id BIGINT,
    upd_cnt INTEGER NOT NULL DEFAULT 1,
    del_flg BOOLEAN NOT NULL DEFAULT false,
    del_date TIMESTAMP,
    del_user_id BIGINT,

    -- foreign key constraints --
    CONSTRAINT fk_opportunity_customer FOREIGN KEY (customer_id)
        REFERENCES cmn_m_customer(id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE,

    CONSTRAINT fk_opportunity_sales_user FOREIGN KEY (sales_user_id)
        REFERENCES cmn_m_user(id)
        ON DELETE RESTRICT
        ON UPDATE CASCADE,

    -- check constraints --
    CONSTRAINT chk_opportunity_amount CHECK (amount >= 0),
    CONSTRAINT chk_opportunity_probability CHECK (probability BETWEEN 0 AND 100),
    CONSTRAINT chk_opportunity_status CHECK (status IN ('OPEN', 'WON', 'LOST', 'PENDING'))
);

-- indexes on foreign keys and filtered columns
CREATE INDEX idx_sfa_t_opportunity_customer ON sfa_t_opportunity(customer_id);
CREATE INDEX idx_sfa_t_opportunity_sales_user ON sfa_t_opportunity(sales_user_id);
CREATE INDEX idx_sfa_t_opportunity_status ON sfa_t_opportunity(status);
```

**Cascading Rule Decision** | カスケードルール選択 | Quyết Định Quy Tắc Cascade:
```
ON DELETE CASCADE:  Use when child rows are meaningless without parent
                    Example: order_items → orders (delete items when order deleted)

ON DELETE RESTRICT: Use when deletion should be prevented if references exist
                    Example: cmn_m_customer → sfa_t_opportunity (prevent customer deletion)

ON DELETE SET NULL: Use when child can exist without parent
                    Example: manager_id → employees (set NULL when manager deleted)
```

**Confidence** | 信頼度 | Độ Tin Cậy: 94%

#### 4. Data Type Selection | データ型選択 | Lựa Chọn Kiểu Dữ Liệu

**Evidence**: PostgreSQL Data Type Documentation

**Data Type Decision Tree** | データ型決定ツリー | Cây Quyết Định Kiểu Dữ Liệu:
```
Primary Key:
  - Project standard → BIGINT GENERATED BY DEFAULT AS IDENTITY
  - Or equivalently → BIGSERIAL

Text:
  - Limited length (name, email)? → VARCHAR(n)
  - Unlimited (description, content)? → TEXT

Numbers:
  - Money/amounts? → DECIMAL(15, 2)
  - Auto-increment ID? → BIGINT GENERATED BY DEFAULT AS IDENTITY
  - Quantity/Count? → INTEGER or BIGINT
  - Version counter? → INTEGER (upd_cnt)

Date/Time:
  - Instant/timestamp? → TIMESTAMP (stored as UTC, converted by R2dbcCustomConverters)
  - Date only? → DATE

Boolean:
  - True/False? → BOOLEAN (not TINYINT, not VARCHAR)
  - Soft delete? → BOOLEAN DEFAULT false (del_flg)

Flexible data:
  - Variable attributes? → JSONB (with GIN index)
  - Tags/metadata? → JSONB
```

**Good Example** | 良い例 | Ví Dụ Tốt:
```sql
CREATE TABLE cmn_m_product (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    product_code VARCHAR(50) NOT NULL UNIQUE,
    product_name VARCHAR(255) NOT NULL,
    description TEXT,
    price DECIMAL(15, 2) NOT NULL,
    quantity INTEGER NOT NULL DEFAULT 0,
    is_active BOOLEAN NOT NULL DEFAULT true,
    attributes JSONB,

    -- audit columns --
    ins_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    ins_user_id BIGINT NOT NULL,
    upd_date TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    upd_user_id BIGINT,
    upd_cnt INTEGER NOT NULL DEFAULT 1,
    del_flg BOOLEAN NOT NULL DEFAULT false,
    del_date TIMESTAMP,
    del_user_id BIGINT,

    CONSTRAINT chk_price CHECK (price >= 0),
    CONSTRAINT chk_quantity CHECK (quantity >= 0)
);
```

**Confidence** | 信頼度 | Độ Tin Cậy: 95%

#### 6. Table Partitioning (Range) | テーブルパーティショニング（範囲） | Phân Vùng Bảng (Theo Phạm Vi)

**Evidence**: PostgreSQL Partitioning Documentation

**When to Partition** | パーティショニングする時 | Khi Nào Phân Vùng:
- Table size >100GB
- Query performance degraded due to table size
- Most queries filter by specific column (ins_date, status, region)
- Need to archive old data (drop old partitions)

**Pattern** | パターン | Mẫu:
```sql
-- Create partitioned table (parent table)
CREATE TABLE sfa_t_activity_partitioned (
    id BIGINT NOT NULL,
    customer_id BIGINT NOT NULL,
    activity_type VARCHAR(50) NOT NULL,
    description TEXT,
    ins_date TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    ins_user_id BIGINT NOT NULL,

    PRIMARY KEY (id, ins_date), -- Partition key must be in primary key

    CONSTRAINT chk_activity_type CHECK (activity_type IN ('CALL', 'VISIT', 'EMAIL', 'MEETING'))
) PARTITION BY RANGE (ins_date);

-- Create monthly partitions
CREATE TABLE sfa_t_activity_2025_01 PARTITION OF sfa_t_activity_partitioned
    FOR VALUES FROM ('2025-01-01') TO ('2025-02-01');

CREATE TABLE sfa_t_activity_2025_02 PARTITION OF sfa_t_activity_partitioned
    FOR VALUES FROM ('2025-02-01') TO ('2025-03-01');

-- Default partition for future data
CREATE TABLE sfa_t_activity_default PARTITION OF sfa_t_activity_partitioned
    DEFAULT;

-- Indexes on partitioned table (inherited by partitions)
CREATE INDEX idx_sfa_t_activity_customer ON sfa_t_activity_partitioned(customer_id);
CREATE INDEX idx_sfa_t_activity_type ON sfa_t_activity_partitioned(activity_type);
```

**Confidence** | 信頼度 | Độ Tin Cậy: 90%

### Rejected Patterns | 拒否パターン | Mẫu Bị Từ Chối

#### ❌ REJECTED 1: Missing Foreign Key Constraints

**Severity** | 重大度 | Mức Độ Nghiêm Trọng: HIGH

```sql
-- ❌ BAD: No foreign key constraint
CREATE TABLE sfa_t_opportunity (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer_id BIGINT, -- No constraint, orphaned records possible!
    sales_user_id BIGINT -- No constraint, deleted users remain referenced!
);

-- ✅ GOOD: With foreign key constraints
CREATE TABLE sfa_t_opportunity (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    customer_id BIGINT NOT NULL,
    sales_user_id BIGINT NOT NULL,

    CONSTRAINT fk_opportunity_customer FOREIGN KEY (customer_id)
        REFERENCES cmn_m_customer(id) ON DELETE RESTRICT,

    CONSTRAINT fk_opportunity_sales_user FOREIGN KEY (sales_user_id)
        REFERENCES cmn_m_user(id) ON DELETE RESTRICT
);
```

**Why Rejected** | 拒否理由 | Lý Do Từ Chối:
- ❌ Orphaned records (data integrity violations)
- ❌ Difficult to maintain referential integrity
- ❌ **Fix**: Always add foreign key constraints

---

#### ❌ REJECTED 2: Using TEXT without Length Limit

**Severity** | 重大度 | Mức Độ Nghiêm Trọng: MEDIUM

```sql
-- ❌ BAD: TEXT for email (should be limited)
CREATE TABLE cmn_m_customer (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email TEXT, -- Allows 1GB of text (unnecessary)
    customer_name TEXT   -- Allows 1GB of text (waste of space)
);

-- ✅ GOOD: VARCHAR with appropriate length
CREATE TABLE cmn_m_customer (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    email VARCHAR(255) NOT NULL,
    customer_name VARCHAR(255) NOT NULL,
    description TEXT -- OK for long content
);
```

**Why Rejected** | 拒否理由 | Lý Do Từ Chối:
- ❌ No length validation (user can insert gigabytes)
- ❌ Wastes disk space and memory
- ❌ **Fix**: Use VARCHAR(n) for limited-length fields

#### ❌ REJECTED 3: UUID as Primary Key

**Severity** | 重大度 | Mức Độ Nghiêm Trọng: HIGH

**Why Rejected** | 拒否理由 | Lý Do Từ Chối:
- ❌ Not consistent with project convention (all 87 entities use Long IDs)
- ❌ Larger storage (16 bytes vs 8 bytes)
- ❌ Slower JOIN performance than BIGINT
- ❌ **Fix**: Use `BIGINT GENERATED BY DEFAULT AS IDENTITY` for all tables

## ⚠️ VIOLATION PATTERNS

**Pattern 1**: Missing foreign key constraints
```regex
REFERENCES.*(?!.*FOREIGN KEY)
```
**Severity**: HIGH
**Message**: "Add explicit FOREIGN KEY constraint with ON DELETE rule"

---

**Pattern 2**: TEXT for limited-length fields
```regex
email\s+TEXT|name\s+TEXT
```
**Severity**: MEDIUM
**Message**: "Use VARCHAR(n) instead of TEXT for limited-length fields"

---

**Pattern 3**: UUID as primary key
```regex
id\s+UUID\s+PRIMARY KEY
```
**Severity**: HIGH
**Message**: "Use BIGINT GENERATED BY DEFAULT AS IDENTITY instead of UUID for primary keys (project standard)"

---

**Pattern 4**: Missing audit columns
```regex
CREATE TABLE\s+\w+\s*\([^)]*\)(?!.*del_flg)
```
**Severity**: HIGH
**Message**: "All tables must include the 8 standard audit columns from AbstractAuditingEntity"
