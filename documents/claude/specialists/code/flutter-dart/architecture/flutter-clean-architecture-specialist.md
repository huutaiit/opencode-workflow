## Architecture: Dependency Rules

| From | Can Import | Cannot Import |
|------|-----------|--------------|
| Domain | `dart:core`, `dart:async` (pure Dart ONLY) | Data, Presentation, Core, `package:flutter/` |
| Data | Domain (entities, repository interfaces) | Presentation |
| Presentation | Domain (entities, usecases via DI) | Data (direct — ONLY via Domain interfaces + DI) |
| Core | `dart:core`, Flutter SDK, third-party packages | Features (direct — core is shared kernel) |

**FORBIDDEN** (DEVIATION_DETECTED — hard block):

```
❌ Domain → import 'package:flutter/'
   Domain must be pure Dart — no Flutter SDK dependency.
   
❌ Domain → Data
   Domain NEVER knows about implementation details (datasources, models, repo impls).
   
❌ Domain → Presentation
   Domain NEVER knows about UI (widgets, pages, BLoCs).
   
❌ Presentation → Data (direct)
   Presentation accesses data ONLY via Domain repository interfaces + DI injection.
   Example: BLoC receives UseCase via constructor, UseCase calls repository interface,
   DI provides RepositoryImpl at runtime.
   
❌ Feature A → Feature B
   Cross-feature imports are FORBIDDEN. Communication via:
   1. Shared entities in core/
   2. Event bus pattern
   3. Shared core services
   
❌ Data → Presentation
   Data layer NEVER references UI widgets, pages, or state managers.
```

## Architecture: File Type Mapping

| File Type | Layer | Path Pattern | Naming Convention |
|-----------|-------|-------------|-------------------|
| Entity | Domain | `features/{f}/domain/entities/{name}.dart` | `{Name}` (PascalCase, Equatable) |
| Repository Interface | Domain | `features/{f}/domain/repositories/{name}_repository.dart` | `{Name}Repository` (abstract class) |
| Use Case | Domain | `features/{f}/domain/usecases/{action}_{entity}.dart` | `{ActionEntity}` (e.g., `GetUser`, `CreateOrder`) |
| Model | Data | `features/{f}/data/models/{name}_model.dart` | `{Name}Model` (toEntity/fromEntity/fromJson/toJson) |
| Remote DataSource | Data | `features/{f}/data/datasources/{name}_remote_data_source.dart` | `{Name}RemoteDataSource` (abstract + impl) |
| Local DataSource | Data | `features/{f}/data/datasources/{name}_local_data_source.dart` | `{Name}LocalDataSource` (abstract + impl) |
| Repository Impl | Data | `features/{f}/data/repositories/{name}_repository_impl.dart` | `{Name}RepositoryImpl` (implements interface) |
| BLoC | Presentation | `features/{f}/presentation/bloc/{name}_bloc.dart` | `{Name}Bloc` (extends Bloc<Event,State>) |
| Cubit | Presentation | `features/{f}/presentation/bloc/{name}_cubit.dart` | `{Name}Cubit` (extends Cubit<State>) |
| State | Presentation | `features/{f}/presentation/bloc/{name}_state.dart` | `{Name}State` (sealed class / @freezed) |
| Event | Presentation | `features/{f}/presentation/bloc/{name}_event.dart` | `{Name}Event` (sealed class) |
| Page | Presentation | `features/{f}/presentation/pages/{name}_page.dart` | `{Name}Page` (StatelessWidget) |
| Widget | Presentation | `features/{f}/presentation/widgets/{name}_widget.dart` | `{Name}Widget` |
| Theme | Core | `core/theme/app_theme.dart` | `AppTheme` |
| Failure | Core | `core/errors/failures.dart` | `Failure` (sealed class) |
| DI Container | Core | `core/di/injection_container.dart` | `configureDependencies()` |
| Shared Widget | Core | `core/widgets/{name}.dart` | `App{Name}` prefix (e.g., `AppButton`) |
| Config | Core | `core/constants/app_constants.dart` | `AppConstants` |

## Architecture: Feature Completeness

### Rule 1: New Feature → MUST have

- [ ] Entity in `domain/entities/`
- [ ] Repository interface in `domain/repositories/`
- [ ] ≥1 UseCase in `domain/usecases/`
- [ ] Model in `data/models/` (with `toEntity()` + `fromJson()`)
- [ ] ≥1 DataSource in `data/datasources/`
- [ ] Repository impl in `data/repositories/` (implements domain interface)
- [ ] BLoC/Cubit in `presentation/bloc/`
- [ ] ≥1 Page in `presentation/pages/`
- [ ] DI registration in `core/di/injection_container.dart`
- [ ] Route registration in GoRouter config

### Rule 2: Add Action to Existing Feature → MUST have

- [ ] UseCase for the action in `domain/usecases/`
- [ ] BLoC Event + State for the action
- [ ] UI trigger (button/gesture) in page/widget
- [ ] Test for the new UseCase

### Rule 3: Validation

- [ ] Domain has ZERO `import 'package:flutter/'` statements
- [ ] No cross-feature direct imports
- [ ] All repository access via abstract interface (never concrete impl in Presentation)
- [ ] All models have `toEntity()` and `fromJson()` / `toJson()`
- [ ] BLoC/Cubit states are Freezed immutable classes

## Patterns

### Pattern 0.1: Three-Layer Structure

The app is organized into 3 layers per feature, plus a shared `core/` kernel:

```dart
// lib/features/user/domain/entities/user.dart
import 'package:equatable/equatable.dart';

class User extends Equatable {
  final String id;
  final String name;
  final String email;

  const User({required this.id, required this.name, required this.email});

  @override
  List<Object?> get props => [id, name, email];
}
```

```dart
// lib/features/user/domain/repositories/user_repository.dart
import 'package:fpdart/fpdart.dart';
import '../../../core/errors/failures.dart';
import '../entities/user.dart';

abstract class UserRepository {
  Future<Either<Failure, User>> getUser(String id);
  Future<Either<Failure, List<User>>> getUsers();
}
```

```dart
// lib/features/user/domain/usecases/get_user.dart
import 'package:fpdart/fpdart.dart';
import '../../../core/errors/failures.dart';
import '../entities/user.dart';
import '../repositories/user_repository.dart';

class GetUser {
  final UserRepository repository;
  const GetUser(this.repository);

  Future<Either<Failure, User>> call(String id) {
    return repository.getUser(id);
  }
}
```

### Pattern 0.2: Feature Module Isolation

Each feature is a self-contained module with its own domain/data/presentation layers.
Features NEVER import from each other directly.

```dart
// lib/features/user/user.dart — barrel export
export 'domain/entities/user.dart';
export 'presentation/pages/user_list_page.dart';
export 'presentation/pages/user_detail_page.dart';
// DO NOT export data layer — internal implementation
```

### Pattern 0.3: Core Shared Kernel

Shared infrastructure that all features depend on:

```dart
// lib/core/errors/failures.dart
sealed class Failure {
  final String message;
  const Failure(this.message);
}

class ServerFailure extends Failure {
  final int? statusCode;
  const ServerFailure(super.message, {this.statusCode});
}

class CacheFailure extends Failure {
  const CacheFailure(super.message);
}

class NetworkFailure extends Failure {
  const NetworkFailure(super.message);
}

class ValidationFailure extends Failure {
  final String field;
  const ValidationFailure(super.message, {required this.field});
}
```

### Pattern 0.4: main.dart Initialization

```dart
// lib/main.dart
import 'package:flutter/material.dart';
import 'package:flutter_bloc/flutter_bloc.dart';
import 'core/di/injection_container.dart' as di;
import 'core/routing/app_router.dart';
import 'core/theme/app_theme.dart';

void main() async {
  WidgetsFlutterBinding.ensureInitialized();
  await di.configureDependencies();
  
  Bloc.observer = AppBlocObserver(); // Pattern 10.5
  
  runApp(const MyApp());
}

class MyApp extends StatelessWidget {
  const MyApp({super.key});

  @override
  Widget build(BuildContext context) {
    return MaterialApp.router(
      title: 'Enterprise App',
      theme: AppTheme.light,
      darkTheme: AppTheme.dark,
      routerConfig: AppRouter.router,
      localizationsDelegates: AppLocalizations.localizationsDelegates,
      supportedLocales: AppLocalizations.supportedLocales,
    );
  }
}
```

### Pattern 0.5: Cross-Feature Communication

When features need to share data, use core/ services — never direct imports:

```dart
// lib/core/services/auth_service.dart — shared by multiple features
abstract class AuthService {
  Stream<AuthState> get authStateChanges;
  Future<String?> get currentUserId;
}

// Features access via DI — never import another feature's BLoC
// lib/features/order/presentation/bloc/order_bloc.dart
class OrderBloc extends Bloc<OrderEvent, OrderState> {
  final GetOrders _getOrders;
  final AuthService _authService; // Shared core service, not UserBloc
  
  OrderBloc(this._getOrders, this._authService) : super(OrderInitial());
}
```

## MUST DO

- Use feature-first folder structure (not layer-first)
- Every feature has `domain/` + `data/` + `presentation/`
- Domain layer is pure Dart (no Flutter imports)
- Use abstract repository interfaces in domain
- Register all dependencies in `core/di/`
- Use Equatable for entities
- Use Freezed/sealed for states and events
- Use Either<Failure, T> for error handling

## MUST NOT DO

- Put business logic in widgets or pages
- Import Data layer from Presentation (always via Domain + DI)
- Import Feature A from Feature B directly
- Use concrete repository in BLoC (always abstract interface)
- Skip DI registration (no manual instantiation)
- Use mutable state classes (always Freezed/sealed)
- Use `dynamic` type when specific type is known
- Catch all exceptions with generic `catch (e)` — use typed catches
